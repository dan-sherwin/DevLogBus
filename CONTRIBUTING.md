# Contributing

Thanks for looking at DevLogBus. This project is still being shaped toward a
public v1 release, so focused issues and small pull requests are the easiest to
review.

## Development Setup

Requirements:

- Go 1.26.3
- Node 24 or compatible current Node for the embedded UI build
- npm
- `golangci-lint`
- `govulncheck`

The local CI script installs missing Go tools when needed:

```bash
./dev/ci-local.sh
```

That script runs:

- UI dependency install and production build
- `go mod tidy`
- `go build ./...`
- `go vet ./...`
- race-enabled Go tests with coverage
- `golangci-lint`
- `govulncheck`
- `gofmt -s` verification

Run it before opening a pull request when possible.

## Common Local Commands

Start the daemon:

```bash
go run ./cmd/devlogbusd run
```

Open the terminal UI:

```bash
go run ./cmd/devlogbus tui
```

Emit a record:

```bash
go run ./cmd/devlogbus emit --source demo --level warn --message "hello"
```

Build release artifacts:

```bash
VERSION=v0.1.0 ./dev/release-artifacts.sh dist/release
```

Build the Chrome Web Store upload zip for Browser Tap:

```bash
dev/browser-tap-store-package.sh dist/browser-tap-store
```

## Coding Style

- Keep changes focused.
- Follow existing package boundaries.
- Prefer simple Go and explicit error handling.
- Use `gofmt`.
- Do not add broad abstractions for one call site.
- Public packages under `pkg/` should not depend on private application
  bootstrap packages.
- If behavior changes, update the docs in the same pull request.

## UI Changes

The browser UI lives under `internal/devlogbusd/ui`.

Run:

```bash
npm --prefix internal/devlogbusd/ui ci
npm --prefix internal/devlogbusd/ui run build
```

The production UI bundle is embedded into `devlogbusd`, so keep
`internal/devlogbusd/ui/dist` current when UI changes are committed.

## Browser Tap Testing

Browser Tap lives under `extensions/chrome-devlogbus`.

Manual smoke:

1. Start `devlogbusd run`.
2. Open `chrome://extensions`.
3. Load `extensions/chrome-devlogbus` as an unpacked extension.
4. Attach to a normal web page.
5. Trigger console and network activity.
6. Confirm records appear in the DevLogBus browser UI.

The Chrome Web Store package script validates that the upload zip has
`manifest.json` at the archive root and does not contain macOS metadata.

## Release Expectations

Release artifacts are generated by GitHub Actions and by:

```bash
VERSION=v0.1.0 RELEASE_STRICT_VERSION=1 ./dev/release-artifacts.sh dist/release
```

Release artifacts include:

- macOS amd64 and arm64 archives
- Linux amd64 and arm64 archives
- Windows amd64 and arm64 archives
- Browser Tap Chrome Web Store zip
- checksums

Do not change release archive names casually; package-manager docs and formulas
depend on them.

## Security Issues

Do not open public issues containing secrets, private logs, customer data, or
vulnerability details that need coordinated disclosure. Use a private contact
path from the project owner until a formal security policy is published.
